There are no active proposals for mainstream languages’ native support to AOP paradigm



WHY?

AOP Libraries



Built for mainstream languages,
they do not give a rich syntax to describe entities intuitively.

→ custom languages

AOP Custom Languages

  • Great ergonomy while modelling AOP entities
  • Higher learning curve
  • Needs a continuous development to work properly
  • Do not inherit the valid tooling of a mainstream language (IDEs, code suggestions, syntax highlighters, linters…)
  • Do not support multi-paradigm programming, they tipycally provide only AOP abstractions

A middle road:
JaKtA, an internal DSL

JaKtA:
Jason-like Kotlin Agents


Internal DSL implemented in Kotlin

  • Native tools maintained by the community
  • Multi-paradigm support
  • Higher learning curve, leveraging a mainstream language
  • Good ergonomy

Why kotlin?


  • Direct support to internal DSL creation
  • Language with a continuously growing community
  • Direct support to multiplatform software development
  • Multi-paradigm support
